## Phase 5: Add Regression Tests

**CRITICAL**: Once the user has validated that the fix or feature works correctly, add tests to prevent silent regressions. **Do NOT skip this phase** — untested code is a liability.

### 5.1 Determine test scope

Based on the files changed during Phase 4, identify which test categories apply:

| Changed area | Test type | Tool / Framework | Location |
|---|---|---|---|
| **Vue components** (`.vue`) | Unit / Integration | Vitest + @vue/test-utils | `__tests__/` next to component |
| **Composables** (`.ts` in `composables/`) | Unit | Vitest | `__tests__/` next to composable |
| **TypeScript utils** (`.ts` in `utils/`) | Unit | Vitest | `__tests__/` next to util |
| **GraphQL queries/mutations** (API contract) | E2E API | Vitest or dedicated E2E | `__tests__/` or `e2e/` |
| **Rust services** (`rust-*`) | Unit / Integration | cargo test | `src/` (inline `#[cfg(test)]`) |
| **Python services** (stats-worker, extraction) | Unit / Integration | pytest | `tests/` in service dir |
| **Django backend** (`services/backend/`) | **E2E API only** | pytest + requests | `tests/e2e/` in backend |

#### Backend testing constraints (Django multi-tenant)

Unit and integration tests for `services/backend/` are **NOT reliable** due to the multi-tenant database routing setup. However, **E2E API tests are viable** and should be written when backend changes are involved.

E2E API tests work by recreating a **complete tenant lifecycle** ("tenant story") from scratch:
1. Create a tenant organization and database
2. Create users with appropriate roles
3. Execute the sequence of API calls that leads to the state being tested
4. Assert on the final API response

**Key rules for backend E2E API tests:**
- Each test must be **self-contained**: it builds its own tenant context from zero
- The order of API calls matters — respect the logical "story" (e.g., create patient before creating exam)
- Use real HTTP requests against the running backend (not Django test client)
- Clean up tenant databases after tests when possible
- These tests are slow by nature — only test the specific behavior affected by the issue
- When E2E API tests are too complex for the scope of the change, fall back to `test_graphql_query()` from Cirrusdoc MCP as a lightweight validation, and document why full E2E was skipped

### 5.2 Check existing test coverage

Before writing new tests, check what's already tested:
- Look for existing `__tests__/` directories next to the modified files
- Read existing test files to understand patterns, mocking conventions, and naming style
- Use Cirrusdoc `find_references()` to check if the changed code is already covered by other tests

### 5.3 Write tests

Follow these principles:
- **Test the behavior introduced or fixed by this issue**, not pre-existing functionality
- **Mirror existing test patterns** in the codebase (same mocking style, same describe/it structure, same naming)
- **For bug fixes**: write a test that would have caught the bug (the test should fail without the fix and pass with it)
- **For new features**: write tests covering the happy path and the main edge cases
- **For frontend components**: use `@vue/test-utils` mount/shallowMount, mock GraphQL calls with `vi.mock`, and test user interactions
- **For composables**: use the `withSetup()` helper pattern from `apps/shared/vue/__tests__/setup.ts`
- **For E2E API tests**: validate the full request/response cycle including error cases (400, 403, 404)

### 5.4 Run and validate tests

```bash
# Frontend tests (Vitest)
docker-compose exec frontend npx vitest run <path-to-test-file>

# Rust service tests
docker-compose exec <rust-service> cargo test

# Python service tests (non-backend)
docker-compose exec <python-service> pytest <test-path>

# Backend E2E API tests
cd infra/e2e && uv run pytest api/ -v
```

All new tests must pass. If a test fails, fix the test or the implementation — do not skip or comment out failing tests.

### 5.5 Ask user for validation

Present a summary of the tests written:
- Number of test files created/modified
- Number of test cases added
- What scenarios are covered (happy path, edge cases, error cases)
- Any areas deliberately not tested and why

Wait for user confirmation before proceeding.

```

```

Use the conventional commit format. The `(#$ARGUMENTS)` suffix automatically links the commit to the issue on GitHub.
